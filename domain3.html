About This Domain

What Auditors Must Know

Domain Scope

Domain 3 covers the controls and audit considerations that apply when an organisation acquires, builds, or deploys information systems. It accounts for 12% of the CISA exam — roughly 18 questions.

  • System Development Life Cycle (SDLC): Phase-gate controls from feasibility to retirement
  • Acquisition controls: Vendor due diligence, RFP, contracts, software escrow
  • Development methodologies: Waterfall, Agile, DevOps — and how audit approach differs per model
  • Testing types: Unit, integration, UAT, regression, performance, penetration testing
  • Change & release management: CAB, emergency changes, SoD controls
  • Configuration management: CMDB, version control, baseline audits
  • Data migration: Reconciliation controls, parallel operations, rollback planning
  • Post-implementation review (PIR): Evaluating whether system met objectives

The Auditor Mindset

The CISA exam expects you to think as an auditor evaluating controls — not as a developer building systems. Key mindset shifts:

  • Phase-gate thinking: Controls must exist at every transition point, not just at go-live
  • SoD as a red line: The developer who builds must never be the one who deploys to production
  • UAT ownership: Business users — not the IT team — must sign off on user acceptance testing
  • Emergency ≠ no controls: Emergency changes still require approval and mandatory PIR
  • Test environment hygiene: No production data (especially PII) in test environments
  • CMDB is the baseline: Any unauthorised deviation from the CMDB is a finding
  • Reconciliation before cutover: Legacy systems must not be decommissioned until data integrity is confirmed
💡

Exam tip: When a question asks "what should the auditor do FIRST?" — the answer almost always involves reviewing controls, policies, or documentation before taking any action.

The Shift-Left Principle

Security and controls should be embedded early ("shifted left") into the SDLC — not bolted on at the end. An auditor who only reviews controls at go-live is too late. The cost of fixing a defect multiplies by 10x at each phase: a requirements error caught in design costs 10× less than one caught in testing, and 100× less than one found in production.

Development Methodologies

Methodology Comparator

The audit approach must adapt to the development model. Select each methodology to understand how controls are structured differently.

Structure
Sequential Phases

Requirements → Design → Build → Test → Deploy. Each phase completes before the next begins.

Audit Approach
Phase-Gate Reviews

Auditor reviews documentation and sign-offs at each phase gate before progression is approved.

Key Controls
Formal Documentation

Feasibility study, requirements spec, design docs, test plans, UAT sign-off, change control log.

Key Risks
Late Discovery

Requirements errors found only in testing. No working software until late. Inflexible to changing business needs.

Structure
Iterative Sprints

Work delivered in 2–4 week sprints. Requirements evolve. Working software produced continuously.

Audit Approach
Sprint-Level Controls

Auditor reviews sprint planning, Definition of Done (DoD), backlog management, and retrospective outputs. Attend sprint reviews.

Key Controls
DoD & Acceptance Criteria

Every user story must have clear acceptance criteria. Definition of Done must include security review and testing gates.

Key Risks
Documentation Gaps

Agile ≠ no documentation. Risk: insufficient audit trail, scope creep, security deprioritised in sprint planning, no formal UAT.

Structure
Continuous Delivery

Code moves from commit → build → test → deploy automatically. Multiple releases per day possible.

Audit Approach
Pipeline Controls

Auditor reviews the CI/CD pipeline configuration, automated gate controls, IaC templates, and deployment approval mechanisms.

Key Controls
Automated Gates

SAST/DAST in pipeline, mandatory code review approvals, automated security scans, environment separation, immutable infrastructure.

Key Risks
Speed vs. Control

Risk of bypassing approval gates, developers deploying directly to production, insufficient testing coverage, no change management process.

Software Acquisition Controls

When an organisation buys rather than builds, the auditor evaluates whether appropriate controls govern the acquisition process.

  • Feasibility study: Business case, cost-benefit analysis approved before procurement
  • RFP / vendor evaluation: Formal scoring criteria, reference checks, financial stability
  • Contract terms: SLA, IP ownership, data ownership, right-to-audit clause, escrow provisions
  • Software escrow: Source code deposited with third party — released if vendor goes bankrupt
  • COTS evaluation: Customisation risks, vendor roadmap alignment, licence compliance
  • Open source risks: Licence obligations (GPL), vulnerability management, no vendor support

Outsourced Development Controls

Third-party developers introduce risks around IP, quality, and security that require specific controls.

  • Background checks: Vendor staff vetting proportionate to data sensitivity
  • IP ownership clause: Contract must explicitly assign IP to the client organisation
  • No production access: Outsourced developers must never access live production data
  • Code review: Client team must review all code before acceptance — no black-box delivery
  • Non-disclosure agreement: NDA covering all confidential data and business logic
  • Right-to-audit: Contract must include auditor's right to inspect the vendor's environment
  • Acceptance testing: Client (not vendor) defines and executes acceptance tests

Critical Exam Distinction

UAT is performed by business users — not the auditor and not the IT team. The auditor reviews whether UAT was performed, whether acceptance criteria were defined, and whether formal sign-off was obtained. The auditor does not perform UAT on behalf of the business.

🔧 Development (DEV)

Where code is written and unit-tested. Must be isolated from production data.

  • Developers have write access
  • Highly volatile — frequent changes
  • No production data permitted
  • Typically uses mock/synthetic data

🧪 Test / Staging (TEST)

Mirrors production as closely as possible for integration, UAT, and performance testing.

  • Business users access for UAT
  • Should replicate prod configuration
  • No PII / real production data (GDPR compliance)
  • Change-controlled — stable for testing
  • Use anonymised/masked data only

🚀 Production (PROD)

Live environment serving real users. Access is strictly controlled.

  • Developers must NOT have write access
  • All changes via formal change control
  • Segregation of duties: dev ≠ deploy
  • Access logs reviewed regularly
  • Rollback plan required for all changes
⚠️

Common audit finding: Production data (including PII) used in test environments without masking. This violates GDPR Article 25 (Privacy by Design) and is a reportable finding regardless of whether any actual breach occurred.

SAST vs DAST

Two complementary approaches to finding security vulnerabilities in applications:

  • SAST (Static Application Security Testing): Analyses source code without executing it. White-box testing. Catches issues like SQL injection patterns, hardcoded credentials, insecure functions. Run during development pipeline.
  • DAST (Dynamic Application Security Testing): Tests the running application from the outside. Black-box testing. Simulates attacker inputs against a live (test) system. Run against deployed app in test environment.
  • SCA (Software Composition Analysis): Scans third-party dependencies and open-source libraries for known CVEs.
  • IAST (Interactive): Combines SAST and DAST — agent embedded in running app during testing.

OWASP Top 10 (2021)

The most critical web application security risks — commonly referenced in CISA exam questions:

  • A01 — Broken Access Control (most critical)
  • A02 — Cryptographic Failures (sensitive data exposure)
  • A03 — Injection (SQL, NoSQL, command injection)
  • A04 — Insecure Design (missing threat modelling)
  • A05 — Security Misconfiguration
  • A06 — Vulnerable Components (unpatched libraries)
  • A07 — Auth Failures (weak passwords, session management)
  • A08 — Software & Data Integrity Failures
  • A09 — Security Logging Failures
  • A10 — SSRF (Server-Side Request Forgery)

Change Management

🔀 Change Routing Decision Tree

Change Categories

  • Standard Change: Pre-approved, low-risk, follows documented procedure. No CAB required. E.g., routine password reset, scheduled patch from approved list.
  • Normal Change: Planned change requiring full CAB review and approval before implementation. Majority of changes fall here.
  • Emergency Change: Unplanned, urgent fix to restore service or prevent imminent harm. Expedited approval (emergency CAB or designated authority). Mandatory PIR after completion.

Change Advisory Board (CAB)

The CAB reviews and approves Normal changes. Membership should include representatives from:

  • IT operations and infrastructure
  • Application development
  • Information security
  • Business stakeholders (affected areas)
  • Service management
💡

Exam tip: Emergency changes must still be approved — by a pre-designated emergency authority, not just implemented without oversight. And they always require a mandatory PIR.

7-Step Change Control Process

Step 1
Request & Log

Formal RFC submitted with impact assessment, rollback plan, and business justification.

Step 2
Assess & Categorise

Change manager classifies as Standard, Normal, or Emergency and assigns priority.

Step 3
CAB Review

For Normal changes: CAB evaluates risk, resource requirements, and schedule conflicts.

Step 4
Approve / Reject

Formal approval (or rejection with reasons) documented before any implementation work begins.

Step 5
Implement

Change implemented in designated maintenance window with rollback plan ready to execute.

Step 6
Test & Verify

Post-implementation testing confirms change achieved intended outcome without side effects.

Step 7
Close & Update CMDB

RFC closed, CMDB updated to reflect new configuration baseline. For emergency: PIR conducted.

Segregation of Duties (SoD) in Change Management

SoD is the most critical control in change management. The CISA exam frequently tests SoD violations as audit findings:

Section 7

Configuration & Release Management

Configuration Management & CMDB

The Configuration Management Database (CMDB) is the authoritative record of all Configuration Items (CIs) in the IT environment — hardware, software, services, and their relationships.

  • Baseline configuration: Approved, documented state of a CI. Any deviation without a change record is a finding.
  • Configuration audit: Auditor compares actual system state against CMDB — differences indicate unauthorised changes.
  • CI relationships: CMDB maps dependencies — if server X goes down, which applications are affected?
  • Version control: All code, scripts, and configurations stored in version control (e.g., Git). Enables rollback to any prior state.
  • CMDB accuracy: CMDB must be updated at change closure — not "whenever we get around to it." Stale CMDB is a control gap.
  • Automated discovery: Tools that auto-scan the environment and update CMDB reduce human error.

Release Management

Release management governs the packaging, scheduling, and deployment of approved changes into production.

  • Release package: All code, configuration, scripts, and documentation bundled together for deployment.
  • Release calendar: Scheduled maintenance windows — protects critical business periods (e.g., month-end close).
  • Go/No-Go decision: Formal checkpoint where all stakeholders confirm readiness before deployment proceeds.
  • Rollback plan: Every release must have a tested rollback procedure. "We'll figure it out if it fails" is not acceptable.
  • Deployment verification: Post-deployment smoke tests confirm the release succeeded before the maintenance window closes.
  • Release notes: Document what changed, what was tested, who approved, and known issues.
Key Audit Question

Can the organisation demonstrate, for any given release: who approved it, when it was tested, what the rollback procedure is, and that the CMDB was updated afterward? If any of these cannot be evidenced — that is a finding.

Real-World Scenarios

Choose the correct auditor response, then read the expert explanation.

Scenario 1 — Agile Development

The Missing Acceptance Criteria

During a sprint review, you observe that user stories are being moved to "Done" status without documented acceptance criteria. The Scrum Master argues that "the team knows what done means" and that formal documentation slows them down. The product is handling customer financial data and is subject to PCI-DSS.

What is the auditor's MOST appropriate response?

The auditor must raise a formal finding. Agile does not mean no controls — it means different controls. For a PCI-DSS environment specifically, every user story touching cardholder data must have explicit, documented acceptance criteria that can be evidenced to external auditors. The Definition of Done must be formalised and must include security requirements. The Scrum Master's argument that "the team knows" is not auditable and creates a control gap. The auditor should also verify that the Definition of Done includes security review, code review, and testing completion — not just functional delivery. Recommending a methodology switch is outside the auditor's role and scope.

Scenario 2 — Emergency Change

The Midnight Hotfix

A critical production bug is causing transaction failures at 11pm on a Friday. The lead developer patches the production database directly to restore service — no change request was raised, no CAB approval was sought, and no rollback plan was documented. Service is restored within 30 minutes. The development manager argues this was justified given the urgency.

As the IS auditor reviewing this incident, what is your conclusion?

This scenario contains multiple findings: (1) No approval obtained — even emergency changes require a pre-authorised emergency approver or emergency CAB sign-off, not zero approval. (2) SoD violation — the developer accessed and modified the production database directly. Developers must not have write access to production. (3) No rollback plan — if the patch had made things worse, there was no way to recover safely. (4) CMDB not updated — the configuration baseline is now inaccurate. (5) No mandatory PIR — all emergency changes require a post-implementation review. The successful outcome does not mitigate these control failures — it was simply fortunate. The auditor should recommend that an emergency change process be formally documented, including pre-approved emergency approvers and mandatory PIR triggers.

Scenario 3 — Data Migration

The Premature Decommission

An organisation migrated from a legacy ERP to a new cloud-based system over a weekend. On Monday morning, the legacy system was shut down. Users immediately report that historical purchase orders from the past 3 years cannot be found in the new system. IT confirms the data was "probably migrated" but no reconciliation report was produced before go-live. The legacy system backups were also overwritten during the weekend migration.

What should the auditor identify as the PRIMARY control failure?

The primary control failure is decommissioning the legacy system before reconciliation was completed and signed off. A data migration must include: (1) Pre-migration data cleansing and mapping, (2) A reconciliation report comparing record counts, key totals, and data integrity checks between source and target, (3) Business sign-off on reconciliation before the legacy system is decommissioned, (4) Preservation of the legacy system (at minimum in read-only mode) for a retention period. Additional findings include: overwriting backups before migration was verified (a catastrophic risk), no parallel run period, and no rollback plan. The "probably migrated" comment from IT demonstrates that data completeness was never formally confirmed — a fundamental audit finding. The auditor should recommend data recovery attempts from any available sources and a root cause analysis of the reconciliation failure.

✓ Known

SDLC

The structured process that defines how information systems are planned, created, tested, and deployed.

The System Development Life Cycle is the formal process governing all phases of system development — from feasibility and requirements through design, build, test, implementation, and post-implementation review. Each phase has defined deliverables and phase-gate controls.

🎯 Exam: Auditors review SDLC controls at each phase gate — not just at go-live. The key question at every gate is: was there documented approval before moving to the next phase?
✓ Known

UAT (User Acceptance Testing)

The final testing phase where business users verify the system meets their requirements before go-live.

UAT is performed by the business owners and end users — not by IT or the auditor. Users verify that the system meets the requirements they specified. Formal UAT sign-off documents are required before any system can move to production. UAT must use realistic test scenarios drawn from actual business processes.

🎯 Exam: If asked who performs UAT — always: business users. If IT performs UAT on behalf of business — that is a finding.
✓ Known

Regression Testing

Re-running all existing tests after a change to verify that previously working functionality still works.

Every time a system change is made — bug fix, patch, enhancement — regression testing must be performed to ensure the change did not break any existing functionality. This is often automated in CI/CD pipelines. Without regression testing, fixes frequently introduce new defects (known as "regression bugs").

🎯 Exam: Regression testing is mandatory after every change — not optional. Skipping regression testing to save time is a control weakness, especially in regulated environments.
✓ Known

CMDB

Configuration Management Database — the authoritative record of all IT assets and their configurations.

The CMDB stores Configuration Items (CIs) — hardware, software, services, and their relationships. The CMDB represents the approved baseline; any deviation without a corresponding change record is an unauthorised change. Auditors compare the CMDB against actual system state to identify undocumented changes. The CMDB must be updated at the close of every change.

🎯 Exam: If the CMDB does not match the actual environment — that is a finding regardless of whether the undocumented change was harmful.
✓ Known

CAB (Change Advisory Board)

The cross-functional group that reviews and approves Normal changes before implementation.

The CAB includes representatives from IT operations, development, security, and business stakeholders. It evaluates change requests for risk, resource requirements, scheduling conflicts, and business impact before granting approval. Standard changes bypass CAB (pre-approved). Emergency changes use an expedited process (Emergency CAB or designated authority). All changes still require some form of approval.

🎯 Exam: Know the three change categories and which requires full CAB: Standard (pre-approved, no CAB), Normal (full CAB), Emergency (expedited approval, mandatory PIR after).
✓ Known

Parallel Testing

Running old and new systems simultaneously and comparing outputs to validate the new system's accuracy.

Parallel testing provides the highest assurance for data migrations — both systems process the same real transactions and results are compared. Any discrepancy is investigated before cutover. It is expensive (requires double the processing capacity) and operationally complex, but is expected for high-risk system replacements such as financial or ERP systems.

🎯 Exam: Parallel testing is the MOST reliable migration validation method. It is also the most expensive. The auditor should recommend it for high-risk migrations.
✓ Known

SAST / DAST

Static and dynamic application security testing — two complementary approaches to finding vulnerabilities.

SAST (Static) analyses source code without executing it — white-box testing, performed during development. Catches issues like hardcoded credentials, SQL injection patterns, insecure API calls. DAST (Dynamic) tests the running application from the outside — black-box testing, performed against a deployed test environment. Simulates real attacker inputs.

🎯 Exam: SAST = code review / white-box. DAST = runtime / black-box. Both should be embedded in the CI/CD pipeline. Neither replaces penetration testing by an independent party.
✓ Known

SoD (Segregation of Duties)

No single individual should be able to initiate, approve, and complete a sensitive transaction alone.

In the development context: the person who writes code must not be the person who deploys it to production. This prevents a developer from introducing malicious code and deploying it without oversight. SoD also applies to change management: the person who requests a change cannot be the person who approves it.

🎯 Exam: A developer with production deployment rights is always a finding. Compensating controls (e.g., detailed logging, mandatory management review) can reduce risk but do not eliminate the SoD concern.
✓ Known

PIR (Post-Implementation Review)

A structured evaluation of whether a system or change achieved its intended objectives after go-live.

PIR assesses: requirements fulfillment, performance vs. SLA, security control implementation, user adoption and training effectiveness, and documentation completeness. PIR is mandatory after all emergency changes. For major system implementations, PIR typically occurs 30–90 days post go-live to allow enough operational data to evaluate performance.

🎯 Exam: If a question asks what should happen after an emergency change — the answer always includes a mandatory PIR. PIR is also an auditor's tool for evaluating whether systems met their business case.
✓ Known

Stress Testing

Testing a system beyond its rated capacity to determine failure behaviour and breaking point.

Stress testing deliberately pushes a system past its designed limits to understand how it fails. The goal is to identify failure modes, data integrity risks during failure, and recovery behaviour. Key questions: Does the system fail gracefully? Does it corrupt data? How long does recovery take? This is different from load/performance testing, which validates behaviour under expected peak load.

🎯 Exam: Stress testing = beyond capacity limits. Load/performance testing = expected peak load. An auditor reviewing BCP/DRP should confirm stress testing results are used to define recovery time expectations.
✓ Known

Definition of Done (DoD)

In Agile, the agreed set of criteria that must be met before a user story is considered complete.

The DoD is the Agile equivalent of phase-gate controls. It must include: code review completed, unit tests passing, integration tests passing, security review completed (SAST scan), documentation updated, and acceptance criteria verified by the product owner. An auditor evaluates whether the DoD is formally documented, consistently applied, and includes security requirements — not just functional delivery.

🎯 Exam: If the DoD does not include security requirements in a regulated environment — that is a finding. "The team knows what done means" is not auditable.
✓ Known

Software Escrow

An arrangement where source code is held by a neutral third party, released if the vendor fails.

If an organisation relies on vendor-developed software and the vendor goes bankrupt or ceases to support the product, software escrow ensures the organisation can access the source code to maintain or modify it. The escrow agreement specifies: deposit frequency, release conditions, verification procedures (is the code actually compilable?), and who holds the escrow.

🎯 Exam: Software escrow is a required control for critical vendor-supplied systems. The auditor should verify the escrow exists, is current, and that release conditions are defined and tested.
✓ Known

Data Reconciliation

The process of verifying that migrated data in the target system matches the source system exactly.

Reconciliation controls include: record count matching (same number of rows), financial total matching (sum of key fields), hash total verification, and spot-check validation of individual records. Reconciliation must be completed and formally signed off before the legacy system is decommissioned. "Probably migrated" is not acceptable evidence.

🎯 Exam: No legacy system should be decommissioned before reconciliation is formally completed and signed off. This is the most frequently tested migration control in the CISA exam.
✓ Known

Rollback Plan

A documented, tested procedure to revert a system or change to its previous state if implementation fails.

Every significant change or migration must have a rollback plan that: defines the trigger conditions for rollback, lists all steps to restore the prior state, identifies who authorises the rollback decision, specifies how long rollback takes, and has been tested in the test environment. "We'll figure it out if it fails" is not a rollback plan.

🎯 Exam: An auditor reviewing a change or migration should always ask: Is there a tested rollback plan? If no — that is a finding regardless of whether the change succeeded.
